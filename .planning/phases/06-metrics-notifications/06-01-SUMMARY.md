## Decisions Made
- **groupBy in tenant extension:** The existing tenant isolation extension did not intercept `groupBy` operations. Added it alongside findMany/findFirst/count/aggregate to ensure metrics queries are properly tenant-scoped. This was a pre-existing security gap, not a plan deviation.
- **COUNT(*)::int in raw SQL:** Cast to int instead of bigint to avoid JSON serialization errors with Fastify's response serializer.
- **completedAt field on Demand:** Added as recommended by research (pitfall 6) -- updatedAt is unreliable for tracking when a demand was completed since any subsequent update changes it.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed groupBy missing from tenant extension**
- **Found during:** Task 1 (Schema updates and metrics API endpoints)
- **Issue:** The tenant isolation Prisma extension intercepted findMany, findFirst, findUnique, count, and aggregate but NOT groupBy. Metrics endpoints use groupBy for cost-per-project, avg-time-per-phase, and agent-success-rate, which would have returned data from all tenants.
- **Fix:** Added `operation === "groupBy"` to the read operations interception condition in tenant.ts
- **Files modified:** packages/database/src/tenant.ts
- **Verification:** API build succeeds, groupBy operations now include tenantId filter
- **Committed in:** bd772b5 (Task 1 commit)

**Impact on plan:** Essential security fix. Without this, groupBy-based metrics would leak data across tenants. No scope creep.
