## Decisions Made
- **(prisma as any) cast for notification model:** The tenant extension uses $allModels.$allOperations which strips individual model type information. Accessing `prisma.notification` directly causes TS2339. Cast to `any` is consistent with the existing codebase pattern documented in decision [01-03].
- **Demand update data cast for completedAt:** Same TypeScript limitation -- the extension-wrapped client doesn't expose completedAt in its types despite the field existing in the schema. Used `as any` on the data object.
- **5 notification.create calls (not 4):** Added NOTIF-02 in both Step 3 escalation AND the outer catch block of merge.worker, since both set mergeStatus to needs_human. The plan specified both locations.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] TypeScript compilation errors with tenant extension types**
- **Found during:** Task 1 (Notification API endpoints) and Task 2 (Worker notification emission)
- **Issue:** `request.prisma.notification` and `prisma.notification` both fail TypeScript compilation because the tenant extension's `$allModels.$allOperations` pattern strips individual model type information. Similarly, `completedAt` in demand update data fails type checking.
- **Fix:** Used `(request.prisma as any).notification` in routes and `(prisma as any).notification` in workers. Used `as any` cast on demand update data objects containing completedAt. This is the established codebase pattern per decision [01-03].
- **Files modified:** apps/api/src/routes/notifications.ts, apps/api/src/queues/agent.worker.ts, apps/api/src/queues/merge.worker.ts
- **Verification:** `pnpm --filter api build` succeeds
- **Committed in:** 7ec233d (Task 1), ba9eff5 (Task 2)
