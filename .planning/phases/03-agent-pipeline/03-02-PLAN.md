  <name>Task 1: Discovery agent with prompt builder, requirements extraction, and ambiguity detection</name>
  <files>
    apps/api/src/agents/discovery.agent.ts
  </files>
  <action>
    Create `apps/api/src/agents/discovery.agent.ts`:

    1. Import `executeAgent` from `./base-agent.js`
    2. Import `discoveryOutputSchema` from `@techteam/shared`
    3. Import `z` from `zod` for `z.toJSONSchema()`
    4. Import `prisma` from `@techteam/database` (raw prisma for reading demand/project -- no tenant scope needed for reads within the worker since tenantId is in job data and we're in a trusted context)

    5. Define `buildDiscoveryPrompt(demand, project)` function that constructs a system + user prompt:
       - System context: "You are a software requirements analyst. Analyze the following development demand and produce structured requirements. Be thorough but concise. If any part of the demand is ambiguous, unclear, or missing critical information, add it to the ambiguities array."
       - User prompt includes:
         - Project name and tech stack: `project.name`, `project.techStack`
         - Project repo URL: `project.repoUrl`
         - Demand title: `demand.title`
         - Demand description: `demand.description`
         - Instructions: "Analyze this demand and produce: (1) Functional requirements with acceptance criteria, (2) Non-functional requirements by category, (3) Complexity estimate (S=trivial, M=few days, L=week+, XL=major effort), (4) Any ambiguities that need human clarification before development can begin, (5) A brief summary."

    6. Export `runDiscoveryAgent(params: { demandId: string, tenantId: string, projectId: string, timeout: number })`:
       - Fetch demand and project from database using raw `prisma` (not tenant-scoped) with `findUniqueOrThrow`
       - Build prompt via `buildDiscoveryPrompt(demand, project)`
       - Convert `discoveryOutputSchema` to JSON Schema via `z.toJSONSchema(discoveryOutputSchema)` (Zod 3.24+ feature)
       - Call `executeAgent({ prompt, schema: jsonSchema, timeoutMs: timeout })`
       - Parse output with `discoveryOutputSchema.parse(result.output)` for runtime validation
       - Determine `hasAmbiguities = output.ambiguities.length > 0`
       - Return `{ output, hasAmbiguities, tokensIn: result.tokensIn, tokensOut: result.tokensOut, costUsd: result.costUsd, durationMs: result.durationMs }`

    NOTE: The worker (from Plan 01) handles all database writes (AgentRun creation, Demand updates, stage advancement, job chaining). This agent function is pure: it reads context, calls the AI, and returns results. The worker decides what to do with the results.

    NOTE: If `z.toJSONSchema()` is not available in the installed Zod version, use `zodToJsonSchema` from the `zod-to-json-schema` package as a fallback (install it). Check availability first with a quick test.
  </action>

Agent is a pure function that the worker invokes.

<task type="auto">
  <name>Task 2: Planning agent with prompt builder, task decomposition, and file mapping</name>
  <files>
    apps/api/src/agents/planning.agent.ts
  </files>
  <action>
    Create `apps/api/src/agents/planning.agent.ts`:

    1. Import `executeAgent` from `./base-agent.js`
    2. Import `planningOutputSchema` from `@techteam/shared`
    3. Import `z` from `zod` for `z.toJSONSchema()`
    4. Import `prisma` from `@techteam/database`

    5. Define `buildPlanningPrompt(demand, project, requirements)` function:
       - System context: "You are a senior software architect. Given the requirements from a discovery phase, produce a detailed technical implementation plan. Break down the work into discrete tasks with specific file paths and dependencies between tasks."
       - User prompt includes:
         - Project name: `project.name`
         - Tech stack: `project.techStack`
         - Repo URL and path: `project.repoUrl`, `project.repoPath`
         - Default branch: `project.defaultBranch`
         - Demand title: `demand.title`
         - Demand description: `demand.description`
         - Discovery requirements (the full JSON from discovery phase): `JSON.stringify(requirements, null, 2)`
         - Instructions: "Create a technical implementation plan that: (1) Decomposes the work into discrete tasks, each with a clear description and type (create/modify/delete/test/config), (2) Lists specific files that will be created or modified for each task, (3) Defines dependencies between tasks (which task must complete before another can start), (4) Orders tasks for optimal execution, (5) Identifies risk areas that need extra attention, (6) Provides a brief summary of the overall approach."

    6. Export `runPlanningAgent(params: { demandId: string, tenantId: string, projectId: string, timeout: number })`:
       - Fetch demand from database using raw `prisma.demand.findUniqueOrThrow({ where: { id: demandId } })`
       - Extract requirements from `demand.requirements` (populated by Discovery phase)
       - If `demand.requirements` is null/undefined, throw Error "Cannot run planning without discovery requirements"
       - Fetch project from database
       - Build prompt via `buildPlanningPrompt(demand, project, demand.requirements)`
       - Convert `planningOutputSchema` to JSON Schema
       - Call `executeAgent({ prompt, schema: jsonSchema, timeoutMs: timeout })`
       - Parse output with `planningOutputSchema.parse(result.output)`
       - Return `{ output, tokensIn: result.tokensIn, tokensOut: result.tokensOut, costUsd: result.costUsd, durationMs: result.durationMs }`

    NOTE: Same pattern as Discovery -- pure function, worker handles DB writes and stage advancement. Planning agent reads demand.requirements (set by worker after Discovery completed) and produces the plan JSON. Worker stores result in demand.plan field and advances stage to 'development'.

    NOTE: The planning agent deliberately does NOT receive the full repo file tree in v1. The prompt gives the project's techStack and repoPath as context. Full repo analysis (PLAN-01 "repo structure") can be enhanced in a future iteration by passing a directory listing or using file tools -- but for v1, the tech stack + requirements provide sufficient context for task decomposition.
