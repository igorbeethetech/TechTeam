# Milestone v2.0: Production-Ready UX

**Objetivo:** Tornar o TechTeam 100% user-friendly para uso diário por toda a empresa.
**Premissa central:** O sistema NUNCA faz deploy/merge automático. Sempre gera PR + branch + link de teste. Testers humanos validam antes do merge (feito manualmente no GitHub).

---

## Mudanças Principais

### 1. Fluxo do Pipeline (HOJE vs NOVO)

**HOJE:**
```
Inbox → Discovery → Planning → Development → Testing → Merge(auto) → Done
```
O merge worker faz git merge + push para main automaticamente.

**NOVO:**
```
Inbox → Discovery → Planning → Development → Testing → Review(humano) → Done
```
Após Testing Agent aprovar:
- Demand vai para "Review" (coluna visual no Kanban)
- Mostra: link do PR, nome da branch (copiar), URL de preview, instruções de teste
- Tester humano testa a branch
- Tester clica "Aprovar" → demand vai para Done
- Tester clica "Rejeitar" → demand volta para Development com feedback
- Merge do PR é feito manualmente no GitHub pelo time

### 2. Links de Teste
- **Link do PR:** Já existe, será mais proeminente
- **Preview URL:** Campo configurável por projeto (template com `{branch}`)
  - Ex: `https://preview.vercel.app/git/{branch}`
- **Instruções de teste:** Campo texto por projeto (como testar localmente)
- **Branch name:** Com botão de copiar

### 3. Visibilidade dos Agentes
- Timeline de atividade na demand (todas transições + runs)
- Status em tempo real com indicador visual
- Output estruturado expandível por fase
- Custo e duração por fase

## Fases

### Fase 12: Review Stage + Test Links
**Escopo:** Mudar o fluxo do pipeline para parar no Review humano.

**Backend:**
- Renomear stage `merge` para `review` no enum Prisma (migration)
- OU manter enum `merge` e só mudar display labels (menos invasivo)
- **Decisão: manter enum, mudar labels** - evita migration destrutiva
- Após testing aprovar: mover para stage `merge` (display: "Review") SEM enfileirar merge job
- Adicionar campos ao Project: `testInstructions` (text), `previewUrlTemplate` (string)
- Nova rota API: `POST /api/demands/:id/review` com body `{ action: "approve" | "reject", feedback?: string }`
  - approve: move para `done`, set `completedAt`
  - reject: move para `development`, increment `rejectionCount`, save feedback
- Desativar o auto-queue do merge worker no agent.worker.ts

**Frontend:**
- Renomear "Merge" para "Review" em STAGE_LABELS
- Novo componente `ReviewSection` no demand detail:
  - Card proeminente com: PR link, branch (copy), preview URL (dinâmico), instruções
  - Botões: "Aprovar ✓" (verde) e "Rejeitar ✗" (vermelho, abre textarea para feedback)
- Kanban: coluna "Review" com styling distinto (borda colorida)

**Arquivos:**
- `packages/shared/src/constants/index.ts` - rename stage label
- `packages/shared/src/types/index.ts` - update display names
- `packages/database/prisma/schema.prisma` - add fields to Project
- `apps/api/src/routes/demands.ts` - add review endpoint
- `apps/api/src/queues/agent.worker.ts` - stop auto-queuing merge
- `apps/web/src/components/demands/review-section.tsx` - new component
- `apps/web/src/components/demands/demand-detail.tsx` - integrate review
- `apps/web/src/components/projects/project-form.tsx` - new fields

### Fase 13: Demand Detail Page Overhaul
**Escopo:** Redesenhar a página de detalhe da demand para ser o hub central de informação.

**Layout novo (de cima para baixo):**
1. **Header:** Título + priority badge + stage badge + agent status indicator
2. **Quick Info Bar:** Branch (copy) | PR link | Preview URL | Custo total
3. **Tabs ou Accordion:**
   - **Visão Geral:** Descrição + metadata
   - **Pipeline:** Progress bar interativo + timeline de atividades
   - **Requisitos:** RequirementsView (já existe)
   - **Plano:** PlanView (já existe)
   - **Desenvolvimento:** Code changes, branch, PR, commit
   - **Testes:** Testing report com detalhes
   - **Review:** ReviewSection (da Fase 12)
   - **Logs:** Agent runs com output expandível
4. **Sidebar (desktop):** Status card fixo com ações rápidas

**Componentes novos:**
- `ActivityTimeline` - cronologia de todas as mudanças de stage + agent runs
- `QuickInfoBar` - barra de informações rápidas com copy buttons
- `DemandTabs` - tabs/accordion para organizar seções
- `CopyButton` - botão reutilizável de copiar para clipboard

**Melhorias:**
- Copy buttons em branch name, PR URL, preview URL
- Indicador visual de agent em execução (pulsing dot + "Agente executando Planning...")
- Ícones de status por seção (✓ completo, ⏳ em andamento, ○ pendente)
- Relative timestamps com tooltip mostrando data/hora exata

**Componentes:**

### Fase 15: Agent Transparency & Logs
**Escopo:** Visibilidade total da execução dos agentes.

**Agent Run expandido:**
- Clicar em um agent run expande detalhes:
  - Output estruturado (JSON formatado, syntax highlighted)
  - Para Discovery: mostra requirements encontrados
  - Para Planning: mostra tasks planejadas
  - Para Development: mostra files changed, approach
  - Para Testing: mostra verdict, test results, code quality
- Duração formatada (2m 34s)
- Custo breakdown (tokens in/out → custo)
- Se falhou: error completo com stack trace

**Phase Progress (no demand detail):**
- Stepper visual: Discovery ✓ → Planning ✓ → Development ⏳ → Testing ○ → Review ○
- Cada step clicável, mostra o output daquela fase
- Tempo gasto em cada fase

**Componentes:**
- `AgentRunExpanded` - detalhes expandíveis do agent run
- `PhaseOutput` - viewer de output por fase
- `PhaseStepper` - stepper visual com estados
- `CostBreakdown` - breakdown de custo por fase
